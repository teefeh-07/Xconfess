# Contract Versioning Policy

This policy defines how smart contract versions are managed across the
`xconfess-contracts` workspace.

## Source Of Truth

- Each contract crate version is the `version` field in
  `xconfess-contracts/contracts/<crate>/Cargo.toml`.
- Deployment metadata must include both the crate version and the WASM SHA-256.
- The generated artifacts:
  - `deployments/contract-wasm-manifest.json` (build output)
  - `deployments/<network>.json` (deployment output)

## Scheme

Contract crates follow Semantic Versioning (`MAJOR.MINOR.PATCH`):

- `MAJOR`: breaking external behavior change (function signature, event schema,
  error-code compatibility, storage migration requirement).
- `MINOR`: backward-compatible feature additions or new events/functions that do
  not break existing consumers.
- `PATCH`: backward-compatible fixes and internal behavior corrections with no
  API/event/error-registry break.

## Release Rules

- All crates deployed in one release window must be built from a single git
  commit and single `Cargo.lock` snapshot.
- Builds must use:
  - `cargo build --locked --workspace --release --target wasm32-unknown-unknown`
- Deployments must use artifacts generated by the same build output.
- Never overwrite deployment records manually; regenerate through
  `./scripts/contracts-release.sh`.

## Compatibility Markers

For runtime compatibility, consumers should prefer:

- `get_version()` for contract semantic version details.
- `get_event_schema_version()` and `get_error_registry_version()` for payload and
  error compatibility checks.

When SemVer and runtime markers disagree, runtime markers are the canonical
compatibility contract for downstream consumers.
